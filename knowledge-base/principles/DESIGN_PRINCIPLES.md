# OPT 设计原则知识库

> **说明**：本文件由 KO（知识管理员）维护，记录 OPT 一人设计公司在实践中沉淀的核心设计原则，整合了 Claude Code（51 万行源码）和 Hermes Agent（17K Stars）的工业级架构哲学。所有 Agent 在执行设计相关任务时必须参考。

---

## 一、系统设计原则

### 原则 1：可审计性优于自动化
> "记忆用 Markdown 文件而非向量数据库；技能用结构化文档而非编译代码——用户随时可以查看和编辑 Agent 的'大脑'。"
> —— 来源：Hermes Agent 设计哲学

**含义**：在设计任何系统时，可读性和可审计性比自动化程度更重要。一个人类可以理解和修改的系统，比一个高度自动化但不透明的系统更有价值。

**应用**：所有配置文件使用 Markdown 或 JSON，不使用二进制格式；所有 Agent 的行为逻辑写在 `SOUL.md` 中，而不是硬编码在代码里。

---

### 原则 2：Fail-closed 安全哲学
> "所有安全相关的默认值都是最保守的。工具默认不可并行、默认非只读、权限默认需要确认。"
> —— 来源：Claude Code 安全架构（BashTool 8 层安全检查）

**含义**：当系统不确定该做什么时，选择最安全的选项（什么都不做），而不是尝试猜测用户意图并执行可能有风险的操作。

**应用**：危险操作（删除文件、发布网站、修改配置）必须有明确的确认步骤；Agent 的工具权限默认最小化（`disallowedTools` 列表）。

---

### 原则 3：TAOR 循环执行
> "Think-Act-Observe-Repeat：在执行任何任务前先思考，执行后观察结果，根据结果决定是否重试。"
> —— 来源：Claude Code TAOR 引擎（QueryEngine 4.6 万行核心驱动）

**含义**：Agent 执行任务不是线性的，而是循环的。遇到错误不是直接报错，而是分析错误、调整策略、重试。

**应用**：Builder 在遇到技术错误时，必须先分析错误信息，尝试 3 次不同的解决方案，才能上报给 CoS。

---

### 原则 4：记忆分层存储（CARROS 架构）
> "L0（会话记忆）→ L1（工作区记忆）→ L2（知识库）→ L3（Auto Dream 深度记忆）"
> —— 来源：Claude Code CARROS 记忆系统 + Hermes Agent 记忆架构

**含义**：不同重要程度的信息应该存储在不同层级，避免重要信息被遗忘，也避免知识库被垃圾信息污染。

**应用**：
- **L0**：当前对话的临时信息，不需要持久化。
- **L1**：当前项目的工作状态，存储在 `workspace/` 目录。
- **L2**：跨项目的通用知识，存储在 `knowledge-base/` 目录。
- **L3**：Auto Dream 整理后的精华，由 KO 定期提炼。

---

### 原则 5：工具即能力边界
> "Agent 能做什么完全由其工具集（allowedTools/disallowedTools）决定，没有任何后门。"
> —— 来源：Claude Code 工具隔离机制

**含义**：每个 Agent 的能力边界由其工具集严格定义。没有在工具集中的能力，Agent 不应该尝试通过其他方式绕过。

**应用**：CoS 没有 `file_write` 工具，因此它不能直接写代码；Builder 没有 `config_modify` 工具，因此它不能修改系统配置。

---

## 二、协作原则

### 原则 6：两步触发（requireVisibleAnchor）
> "所有跨 Agent 的协作都必须在人类老板的视线范围内，通过可见锚点 + 执行的两步触发机制完成。"
> —— 来源：Claude Code 多 Agent 协同架构

**应用**：所有 A2A 调用必须在对话中留下 `[A2A] CoS → Builder: 执行任务 #TASK-001` 格式的可见锚点。

---

### 原则 7：QAPS 任务结构
所有跨 Agent 的任务传递，必须遵循 QAPS 结构（Question / Answer / Process / Source），确保任务描述清晰、可验收。

---

## 三、产品设计原则

### 原则 8：拷贝即升级
> 系统的所有状态都应该存储在文件中，而不是存储在运行时内存或数据库中，使得系统可以通过拷贝文件来迁移和升级。

**应用**：所有配置、记忆、知识库都存储在 Git 仓库中；`.env` 文件是唯一需要手动配置的文件。

---

### 原则 9：技能自进化（Hermes 闭环学习）
> "创建技能 → 使用技能 → 发现改进点 → 更新技能 → 下次使用更好的版本"
> —— 来源：Hermes Agent 闭环学习系统

**应用**：Builder 每次成功解决复杂问题后，必须将解决方案沉淀为 `SKILL.md`；KO 定期审核技能库，删除过时的技能。

---

### 原则 10：辩证式用户建模
> "不是简单追加新信息，而是通过正-反-合的辩证融合，生成更准确的用户画像。"
> —— 来源：Hermes Agent Honcho 用户建模机制

**应用**：KO 在更新 `USER_PROFILE.md` 时，必须使用辩证融合，而不是简单追加新观察。
