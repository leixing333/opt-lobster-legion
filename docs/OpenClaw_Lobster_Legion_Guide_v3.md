## 2. 系统基石：记忆与知识管理

一个没有长期记忆的团队是无法持续进化的。OPT 军团采用**三层记忆架构**，确保经验能够沉淀为组织资产。

### 2.1 三层记忆架构

1. **Layer 0：原始对话（短期记忆）**
   - 存在于 Slack/飞书 的聊天记录中。
   - 作用：用于 Ops 审计和上下文追溯。
   - 缺点：容易丢失，检索成本高。

2. **Layer 1：Closeout 摘要（中期记忆）**
   - 存储位置：`workspace-{agentId}/memory/YYYY-MM-DD.md`
   - 机制：每个任务完成后，Agent 必须生成 10-15 行的结构化总结（Closeout）。
   - 作用：压缩上下文，让 Agent 快速回忆起昨天做了什么。

3. **Layer 2：RAG 知识库（长期记忆）**
   - 存储位置：全局 `knowledge-base/` 目录。
   - 机制：由 KO (Knowledge Officer) 定期从 Layer 1 中提炼原则、模式和踩坑记录。
   - 作用：跨项目复用的核心资产。

### 2.2 技能即资产 (Skills as Assets)

在 OPT 军团中，**技能（Skill）是最高级别的知识沉淀**。当某个任务流程被成功执行 3 次以上，KO 会将其提炼为 `SKILL.md` 文件。

例如，当你第三次要求 Builder 将网站部署到 Vercel 时，KO 会自动生成 `opt-web-publisher` 技能。下次再部署时，Builder 只需调用该技能，即可实现一键发布。

## 3. 协同作战：A2A 通信协议

为了防止多个 Agent 在同一个上下文中互相干扰，OPT 军团采用**物理隔离的沙箱环境**和**两步触发协议**。

### 3.1 频道隔离

每个 Agent 绑定一个独立的 Slack 频道（或飞书群组）：
- `#hq`：老板与 CoS 的专属频道。
- `#cto`：技术讨论频道。
- `#build`：代码提交和部署频道。

### 3.2 两步触发 (Two-Step Trigger)

当 CTO 需要 Builder 写代码时，不能直接在后台调用，必须遵循以下步骤：

1. **可见锚点**：CTO 在 `#build` 频道发送一条可见消息：“@Builder，请执行工单 #123”。
2. **触发执行**：CTO 调用 `sessions_send` 工具，将完整任务内容发送给 Builder。

这种机制确保了**所有跨 Agent 的协作都在人类老板的视线范围内**，防止 AI 团队在后台“失控”。

## 4. 持续生产力：安全与备份

一个具备持续生产力的系统，必须是安全且可恢复的。

### 4.1 角色漂移防护

AI Agent 在长时间运行后，容易出现“角色漂移”（例如 Builder 开始做架构设计，或者 CoS 开始写代码）。
- **防线**：Ops 审计员每周会检查各 Agent 的对话摘要，对照其 `SOUL.md` 判断是否越界。
- **修复**：一旦发现漂移，Ops 会建议老板对该 Agent 进行“洗髓”（重新强化系统提示词）。

### 4.2 拷贝即升级 (Copy to Upgrade)

OPT 军团的设计目标之一是**极高的可移植性**。通过 `opt-auto-backup` 技能，系统每天凌晨会将配置文件、工作区和知识库打包备份到 GitHub 私有仓库。

当你更换电脑或服务器时，只需执行以下三步：
1. 安装 OpenClaw。
2. 克隆备份仓库。
3. 运行 `openclaw backup restore`。

15 分钟内，你的整个 AI 团队将满血复活，带着所有的记忆和技能继续为你工作。

## 5. 附录：从 Claude Code 源码泄露学到的顶级 Agent 架构范式

### 5.1 TAOR 引擎：超越简单对话的反馈循环

Claude Code 的核心运行机制是 TAOR 引擎（Think-Act-Observe-Repeat）。它不是一个简单的对话框，而是一个持续运行的反馈循环体。

- **Think**：分析当前状态，理解意图，制定下一步计划。
- **Act**：调用底层原语（如 Read、Write、Execute）。
- **Observe**：读取执行结果，评估是否报错，判断目标是否达成。
- **Repeat**：未完成则调整策略，继续循环。

**在 OPT 中的应用**：我们的 Builder 和 Researcher Agent 严格遵循这一模式。在执行复杂任务（如代码编写或深度调研）时，它们被配置为在遇到错误时自动捕获输出、分析原因并重试，而不是直接向用户报错。

### 5.2 四层记忆系统（CARROS）与 Auto Dream

Claude Code 实现了一套多层级、多作用域的持久化记忆架构：

1. **会话记忆**：实时对话上下文，动态压缩。
2. **工作区记忆**：项目级文件、代码关联，本地持久化。
3. **全局记忆**：跨项目通用知识、用户习惯。
4. **深度记忆（Auto Dream）**：后台记忆巩固层，模拟人类睡眠，在空闲时自动整理、萃取和索引记忆碎片。

**在 OPT 中的应用**：OPT 的记忆系统分为三层（L0 会话、L1 Closeout、L2 RAG）。我们引入了 KO（知识管理员）角色，其核心职责类似于 Auto Dream 机制——定期在后台（或通过定时任务）将 L1 的碎片化总结提炼并向量化存入 L2 知识库，实现知识的长期沉淀。

### 5.3 严格的工具隔离与权责分离

Claude Code 内置了 6 个专业化的 Agent（如 Explore、Plan、Verification 等），每个 Agent 都有明确的 `disallowedTools` 列表。例如，Explore Agent 被严格禁止任何写操作，只能读文件和搜索。

**在 OPT 中的应用**：OPT 的 6 个 Agent 同样实施了严格的工具隔离。
- **Researcher**：仅限使用搜索、网页读取工具，禁止文件写入。
- **Ops**：专注于审计和备份工具，禁止修改业务代码。
- **Builder**：拥有代码编辑权限，但必须在 CTO 的架构指导下进行。
这种 Unix 哲学（一个工具只做一件事）彻底杜绝了 Agent 的角色漂移和越权操作。

### 5.4 Fail-closed 安全默认与操作审计

Claude Code 的所有安全相关默认值都是最保守的（如默认不可并行、默认非只读）。其 BashTool 实现了 8 层安全检查，包括 AST 级命令解析和命令注入检测。
