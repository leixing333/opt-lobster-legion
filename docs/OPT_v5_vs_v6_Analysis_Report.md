# OPT 龙虾军团 v5.0 vs v6.0 深度比较与架构分析报告

## 1. v5.0 与 v6.0 的系统性比较

在对 OPT 龙虾军团 v5.0 和 v6.0 的核心文件进行深度对比后，我们可以清晰地看到 v6.0 在架构深度、角色细分和安全机制上的全面升级。v6.0 并非简单的文件堆砌，而是对三大顶尖开源架构（Claude Code、Hermes Agent、OpenHarness）的深度融合与重构。

### 1.1 核心差异数据对比

| 维度 | v5.0 | v6.0 | 升级意义 |
|---|---|---|---|
| **Agent 数量** | 6 个（CoS, CTO, Builder, Researcher, KO, Ops） | 8 个（新增 FE, BE, QA，拆分 Builder） | 实现了从"全栈单打独斗"到"专业流水线"的转变，降低了单个 Agent 的上下文负担。 |
| **文件总数** | 31 个 | 30 个 | 移除了冗余的 AGENTS.md，将规范内聚到 SOUL.md 和配置文件中，结构更清晰。 |
| **代码/文档行数** | 3004 行 | 3300 行 | 增加了近 300 行的核心逻辑，主要集中在 Hooks 拦截器、Auto Dream 引擎和详细的 SOUL.md 规范中。 |
| **配置文件** | `openclaw.json` (基础配置) | `config/openclaw.json` (113行，含 MCP、Hooks、权限矩阵) | 实现了真正的"代码即 Harness"，所有权限和路由均可配置化。 |
| **安全机制** | 基础的 Fail-closed 概念 | 真实的 Python Hooks 拦截器 (`permission_check.py`, `audit_logger.py`) | 从"概念约束"升级为"代码级硬拦截"，确保系统不会越界。 |

### 1.2 架构理念的演进

在 v5.0 中，系统主要依赖于 Prompt 层面的约束（如 `AGENTS.md`），这在实际运行中容易发生"角色漂移"。而 v6.0 彻底贯彻了 **"The model is the agent. The code is the harness."** 的理念：

1. **认知与执行的彻底解耦**：v6.0 中，Agent 的 SOUL.md 专注于认知（如 FE 专注于 UI/UX，BE 专注于 API），而底层的执行约束交给了 `hooks/` 目录下的 Python 脚本。例如，`permission_check.py` 会硬性拦截 `rm -rf /` 等危险命令，而不需要依赖模型自身的判断。
2. **动态 Swarm 机制**：v6.0 的 CoS 引入了 `/swarm` 命令，可以根据任务动态组建临时团队（如 FE + BE + QA），这比 v5.0 中固定的 A2A 派单更加灵活。

## 2. v6.0 的完整思考过程与架构思路实现原理

v6.0 的设计初衷是为了解决 v5.0 在复杂软件开发场景下的局限性。以下是核心架构思路的实现原理：

### 2.1 为什么拆分 Builder？（降低 Token 污染与上下文漂移）

在 v5.0 中，Builder 承担了所有的编码工作。当项目变大时，前端的 React 组件代码和后端的数据库 SQL 语句会混合在同一个上下文中，导致模型产生"幻觉"或"上下文污染"。

**v6.0 的解决方案**：将 Builder 拆分为 FE（前端）、BE（后端）和 QA（测试）。
- **FE** 专注于视觉一致性和组件开发，其 SOUL.md 强制要求遵循 `DESIGN_SYSTEM.md`。
- **BE** 专注于业务逻辑和数据建模，运行在独立的上下文中。
- **QA** 作为质量守门员，负责编写测试用例，形成闭环验证。
这种拆分使得每个 Agent 的上下文更加纯粹，推理能力更强。

### 2.2 三大架构在 v6.0 中的具体实现位置

1. **Claude Code (TAOR 引擎与 Fail-closed)**：
   - **实现位置**：`engine/taor_loop.sh` 和各 Agent 的 SOUL.md。
   - **原理**：FE 和 BE 在执行构建或测试时，如果遇到错误，不会直接报错给 CoS，而是捕获 stderr，分析错误原因并自动重试（最多 3 次）。Fail-closed 原则体现在 `openclaw.json` 的 `permissions` 配置中，默认拒绝高风险操作。

2. **Hermes Agent (技能自进化与辩证建模)**：
   - **实现位置**：`engine/auto_dream.py` 和 KO 的 SOUL.md。
   - **原理**：每周一凌晨，`auto_dream.py` 会扫描 `memory/session/` 下的会话记录，提取经历了多次 TAOR 重试才成功的"高价值轨迹"，自动生成新的 `SKILL.md` 存入 `skills/evolved/`。同时，KO 使用"正-反-合"逻辑更新用户画像。

3. **OpenHarness (Hooks 机制与 MCP 生态)**：
   - **实现位置**：`hooks/pre_tool_use/`、`hooks/post_tool_use/` 和 `config/openclaw.json`。
   - **原理**：在任何工具调用前，输入数据会通过 stdin 传给 `permission_check.py`，只有返回 `allowed` 才能执行。执行后，输出数据会传给 `audit_logger.py` 进行敏感信息脱敏（如 API Keys）并记录审计日志。MCP 服务器（如 GitHub、Postgres）的配置则直接写在 `openclaw.json` 中。

## 3. 从 v5.0 及 OpenClaw 原研究中挖掘的遗漏功能与下一步规划

在对比 Claude Code 51万行源码的深度分析报告和当前的 v6.0 实现后，我们发现仍有一些高级特性尚未完全实现。以下是遗漏功能清单及下一步开发规划：

### 3.1 遗漏的重要功能清单

1. **KAIROS 守护进程（追加式日志模式）**：
   - **缺失说明**：Claude Code 中有一个 KAIROS 模式，用于长会话助理，采用追加式日志（`logs/YYYY/MM/YYYY-MM-DD.md`）而非覆盖式更新。目前 v6.0 的记忆系统仍偏向于覆盖式。
2. **细粒度的记忆漂移防护（Memory Drift Caveat）**：
   - **缺失说明**：Claude Code 在使用超过 1 天的记忆时，会强制附加警告："This memory is N days old... Verify against current code before asserting as fact." 目前 v6.0 的 KO 尚未实现这种时间衰减警告。
3. **Session Memory 的增量压缩（Compact 替代 LLM 摘要）**：
   - **缺失说明**：Claude Code 的 Session Memory Agent 会在每次采样后更新一份固定结构的笔记，用于在上下文压缩时直接替代昂贵的 LLM 摘要调用。v6.0 目前的 Closeout 流程仍依赖 LLM 进行全量摘要。
4. **多模态文件解析能力**：
   - **缺失说明**：目前系统主要处理文本和代码，缺乏对设计图（UI/UX）的视觉理解能力，这对于 FE 角色来说是一个短板。

### 3.2 下一步开发规划 (v6.1 - v7.0)

- **Phase 1 (v6.1): 记忆系统增强**
  - 在 `engine/auto_dream.py` 中引入时间衰减算法，为旧记忆自动添加 `[WARNING: STALE]` 标签。
  - 实现 Session Memory 的增量更新机制，降低 Closeout 时的 Token 消耗。
- **Phase 2 (v6.2): KAIROS 日志系统**
  - 为 CoS 引入 KAIROS 模式，支持按天生成的追加式工作日志，方便老板随时查阅历史决策。
- **Phase 3 (v7.0): 多模态与 MCP 深度集成**
  - 引入视觉模型支持，允许 FE 直接读取 Figma 导出的图片并生成代码。
  - 扩展 MCP 服务器列表，接入更多外部 SaaS 工具（如 Jira, Slack）。
